# Story 4.3.3: Implement Delta Affinity Calculation Engine

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a recommendation engine developer,
I want to update the RecommendationService to process Delta (Δ) logic that evaluates the distance between a problematic tag and a candidate tag concerning a team's Big Five personality profile,
so that the system can provide highly targeted practice adaptation advice using explicit tag dependencies and personality-affinity delta calculations.

## Acceptance Criteria

1. Given a set of "problematic" or "missing" tags for an issue, when the recommendation engine is invoked, then it queries the `tag_candidates` table to compile a list of Candidate Tags.
2. Given a Candidate Tag and a Problematic Tag, when calculating the Delta score for the team, then it correctly maps the team's average personality affinity transitions to absolute values (see affinity scoring with high/low values and interpolation).
3. Given a Candidate Tag where the absolute team affinity is negative (`-` indicating clear friction), when evaluating recommendations, then that Candidate Tag is strictly disqualified, regardless of a potentially high comparative Delta score.
4. Given the final evaluation, then the service returns a list of candidate tags sorted descending by their Delta score, breaking ties using positive absolute affinity.

## Dev Notes

### Core Algorithm Guardrails

1. Input scope:
   - Use issue-selected source tags from Story 4.3.1 (`problematic` or `missing`) as the "current" side.
   - Resolve candidate tags via `tag_candidates(problem_tag_id -> solution_tag_id)`.

2. Affinity computation reuse (non-negotiable):
   - Reuse the same trait-level piecewise clamp/interpolation model as existing affinity engine (`computeTraitContribution`).
   - Formula to keep consistent with Epic 4.1:

     `contribution = lowEndpoint + ((userValue - lowBound) / (highBound - lowBound)) * (highEndpoint - lowEndpoint)`

   - Boundary handling must stay identical:
     - `userValue <= lowBound` => `lowEndpoint`
     - `userValue >= highBound` => `highEndpoint`
     - interpolation only between bounds
   - Endpoint mapping remains: `'-' -> -1`, `'0' -> 0`, `'+' -> 1`.

3. Team aggregation:
   - Compute per-member tag affinity first (eligible members only), then team average.
   - Never expose raw teammate trait values in recommendation outputs (aggregate only).

4. Delta transition scoring (directed recommendation contract):
   - Delta raw: `candidateAffinity - currentAffinity` (for traceability/debug).
   - Delta gain class (ranking score):
     - `- -> +` => `+1.0`
     - `- -> 0` or `0 -> +` => `+0.5`
     - `0 -> -` => `-0.5`
     - `+ -> -` => `-1.0`
   - Optional neutral same-sign transitions should preserve deterministic behavior (recommend explicit fallback to raw delta for ordering inside same gain class).

5. Hard disqualification rule:
   - If candidate absolute team affinity is `< 0`, reject candidate immediately, even if gain class or raw delta looks positive.

6. Ranking:
   - Primary: Delta score descending.
   - Secondary tie-break: candidate absolute affinity descending (prefer stronger positive fit).
   - Tertiary tie-break: deterministic stable order (ID asc).

### Implementation Guidance

- Current `recommendation.service.ts` implements Story 4.2 practice alternatives; directed tag engine should be additive and must not regress existing behavior.
- Favor batched Prisma reads to avoid N+1 loops when evaluating multiple (problemTag, candidateTag) pairs.
- Deduplicate repeated candidate tags that come from multiple problem tags; keep best scoring pair and preserve provenance list for explainability.
- Keep response structures camelCase in service/API boundaries.

### Security and Privacy Constraints

- Recommendation outputs must include aggregated team affinity context only.
- Do not return or log individual member Big Five trait values.
- Preserve existing structured error shape and requestId propagation pattern.

### Testing Requirements (Minimum Matrix)

- Negative candidate disqualification with otherwise high positive delta.
- Tie-break behavior with equal delta gains.
- No-candidate and all-disqualified fallback behavior.
- Regression protection for Story 4.2 recommendation outputs and UI consumption expectations.
